Genies · under the hood

Cross-platform C++ avatar runtime

One C++ core that builds, composes, and renders avatars natively on iOS, Android, desktop, and the browser. Everything platform-specific lives in a thin wrapper on top, so the avatar behaves identically everywhere.

The problem it solves

The same avatar has to look and behave the same in a Swift app, an Android app, a Unity game, and a web page. Rebuilding that logic four times guarantees the four will drift apart and bug out in different ways. So the whole thing is one C++ engine, and each platform just gets a small binding layer.

How it is put together

Platform wrappers iOS (Obj-C++) · Unity (C#) · Android · WebGL
↑ thin bindings ↑
Binding layer SWIG-generated bindings · Emscripten / WASM build for the web
Core composition & slots · entity system · mesh runtime · materials
Graphics Vulkan / MoltenVK · Metal · OpenGL ES / WebGL · Slang shaders
Serialization FlatBuffers schemas · zero-copy, versioned avatar data
  • Composition & entities. Wearables stack onto a base body through a layered system of slots and rules, built on an entity/attribute model that merges meshes, skeletons, and materials in a deterministic order.
  • Mesh runtime. A shared skeleton with GPU skinning, blendshape (morph target) baking, and a runtime mesh combiner that fuses the body and every wearable into a small set of optimized draw primitives. A rest-cage deformation system refits geometry as the body shape changes.
  • Materials. A full PBR system covering the glTF material extension set (clearcoat, sheen, transmission, iridescence, specular, volume), dynamic and procedural materials, and LOD texture streaming so a phone only pays for the detail on screen.
  • Graphics. One backend-agnostic context with Vulkan (MoltenVK on Apple), Metal, and OpenGL ES / WebGL implementations. Shaders are written once in Slang and cross-compiled; a WebGL GLSL sanitizer smooths over browser quirks; textures use compressed KTX.
  • Serialization & wrappers. FlatBuffers gives zero-copy, versioned avatar data; SWIG bindings and an Emscripten build expose the same core to Unity, iOS, Android, and the web.

Decisions I am glad we made

  • One source of truth in C++, so avatar behavior can't quietly drift between platforms.
  • Abstract the GPU behind a single context, so one render path serves mobile, Apple's Metal, and the browser.
  • FlatBuffers for zero-copy loads, which really matters on memory-constrained phones.
  • Author shaders once in Slang instead of hand-maintaining GLSL, HLSL, and Metal Shading Language in parallel.

My part

I built the runtime and its module layout, the graphics abstraction that hides three GPU APIs behind one interface, and the wrapper and binding layer that ships the same core to every platform the product runs on.

Genies · under the hood

Distributed avatar-generation pipeline

Turning a raw, AI-generated body mesh into a fully rigged, wearable-ready avatar, as a fault-tolerant distributed pipeline instead of one long monolithic job.

The problem it solves

The original autorigger was a single monolithic job that mixed slow, CPU-bound geometry work (Blender and USD) with GPU ML inference (pose estimation and segmentation). You couldn't scale the two kinds of work separately, and a failure anywhere meant redoing the whole thing.

The redesign

I split the monolith at its ML-inference boundaries, so the CPU/USD work and the GPU ML work became separate, independently schedulable stages. The stages never call each other directly. An orchestrator routes between them, and they hand large 3D files back and forth through a shared object-storage working directory keyed by the asset being built.

How the pieces coordinate

  • Message-driven workers. Each stage is a worker that consumes a task message off Kafka and returns a structured result envelope. The orchestrator reads each envelope to decide what runs next, and halts the chain the moment a stage fails.
  • Shared working directory. Every stage syncs its inputs down from object storage, does its work, and syncs outputs back up under the same asset key. The ML workers drop their results into the same place, so the final assemble step just reads them.
  • Clean process isolation. The heavy Blender/USD steps run in fresh subprocesses rather than the worker's event loop, because those libraries are not thread-safe and expect to own a clean process's main thread.

Why it is better

  • CPU geometry stages and GPU ML stages scale independently, on the right hardware.
  • Real concurrency: the hair rigs while the bust is still being processed.
  • Every stage is restartable and observable, instead of one long opaque job.
  • Services talk through stable contracts, so any one piece can change without breaking the chain.

My part

I re-architected the monolith, chose where to cut the stage boundaries, and designed the shared-storage coordination and the worker/orchestrator contracts that hold it all together.

Genies · under the hood

Lens, the internal 3D tooling platform

One platform, frontend and backend, that hosts many internal tools behind a single login, with real-time data from the avatar pipeline and rigged 3D avatars rendered right in the browser.

The problem it solves

The avatar pipeline runs across a constellation of distributed services (the one in the card next door). Engineers and artists needed one place to watch jobs run, inspect what came out, and use internal tooling, instead of tailing logs and poking raw APIs. Lens is that place.

How it is put together

Browser · one React 19 app modular workspaces (job monitor, avatar autogen, avatar viewer, wearables) under one shell
Auth WorkOS AuthKit · OAuth login · JWT rides with every backend call
Realtime data Convex queries / mutations / actions · live job progress over subscriptions, no polling
In-browser 3D Three.js · React Three Fiber · the C++ avatar runtime compiled to WebAssembly
Integrations, behind actions S3 (presigned) · CMS · storage · pipeline jobs — no secrets in the client
  • Workspaces, not separate apps. One authenticated shell provides auth, layout, data, and the 3D viewers; each internal tool is a self-contained "workspace" with its own URL namespace, navbar, and pages. Adding a tool means adding a workspace, not standing up a new app.
  • Real-time by default. Convex pushes job progress over live subscriptions, so the dashboard tracks the pipeline as it runs, each step moving from queued to running to done in front of you, with no refreshing or polling.
  • The runtime, in the browser. The same C++ avatar runtime from the card next door is compiled to WebAssembly and embedded as a viewer, so you can inspect the rigged avatar a job produced without leaving the page. Private assets get presigned first.
  • Secrets stay server-side. Every privileged call (storage, CMS, job data) sits behind a backend action, so the browser never holds a credential.

Why it matters

Lens is where my other two Genies systems become visible and usable: it monitors the distributed pipeline and it renders the avatars the runtime builds. It turns a backend that used to live in logs and dashboards into something the whole team can actually see and drive.

My part

I built Lens end to end: the frontend architecture and the workspace platform that lets new internal tools slot in cleanly, the backend that serves and authenticates it, and the real-time job monitoring on top. I also own the connective layer that ties Lens into the rest of Genies' stack, wiring it up to the pipeline, storage, and CMS services other teams built, plus the in-browser integration of the C++ avatar runtime as a 3D viewer.
